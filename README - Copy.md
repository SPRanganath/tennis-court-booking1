# CourtSide - Tennis Court Booking

React + Vite frontend, Express API, Neon (serverless Postgres) for
persistence, Better Auth for email/password authentication. Ships as a
single deployable Node process: Express serves both the API and the built
frontend, so there's nothing separate to run in production.

## Setup

1. Install dependencies:
   ```
   npm install
   ```
2. Create a [Neon](https://neon.tech) project (free tier is fine) and copy its
   connection string.
3. Copy `.env.example` to `.env` and fill in:
   - `DATABASE_URL` - your Neon connection string (use `sslmode=verify-full`)
   - `BETTER_AUTH_SECRET` - generate with
     `node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"`
   - `BETTER_AUTH_URL` / `BETTER_AUTH_TRUSTED_ORIGINS` - defaults in
     `.env.example` work for local dev as-is
4. Create the schema (Better Auth's `user`/`session`/`account`/
   `verification` tables, then `courts`/`bookings`) and seed the mock courts:
   ```
   npm run db:migrate
   npm run db:seed
   ```

## Development

```
npm run dev
```

Runs the Vite dev server (with HMR) on `http://localhost:5173` and the
Express API on `http://localhost:3001` at the same time. Vite proxies
`/api/*` requests to the API, so the browser only ever talks to port 5173.

## Production (single artifact)

```
npm run build
npm start
```

`npm run build` outputs the frontend to `dist/`. `npm start` runs one Node
process (`server/index.js`) that serves that static build *and* the `/api/*`
routes on one port (`PORT`, default `3001`). That single process is the
whole deployable artifact - point any Node host (Render, Railway, Fly.io, a
VM, etc.) at `npm run build && npm start` with `DATABASE_URL`,
`BETTER_AUTH_SECRET` and `BETTER_AUTH_URL` set and it's live. In production
`BETTER_AUTH_TRUSTED_ORIGINS` isn't needed since the frontend and API share
one origin.

## Deploying to Render

This repo includes a `render.yaml` Blueprint, so Render can configure the
service automatically:

1. Push this repo to GitHub (or GitLab).
2. In the Render dashboard: **New > Blueprint**, point it at the repo. Render
   reads `render.yaml` and creates a single Node web service with the build
   command (`npm install && npm run build`), start command (`npm start`),
   and health check (`/healthz`) already set.
3. When prompted for env vars, set `DATABASE_URL` to your Neon connection
   string (`sslmode=verify-full`). `BETTER_AUTH_SECRET` is auto-generated by
   the Blueprint. `BETTER_AUTH_URL` doesn't need to be set - the server picks
   up Render's own `RENDER_EXTERNAL_URL` automatically.
4. After the first deploy, run the migration and seed once from your local
   machine against the same `DATABASE_URL` (Render's free plan has no shell
   access):
   ```
   npm run db:migrate
   npm run db:seed
   ```
5. Render gives the service a public `https://<name>.onrender.com` URL -
   that's the whole app, globally reachable, no separate frontend deploy.

Without the Blueprint, the same thing works as a manually-configured Web
Service: build command `npm install && npm run build`, start command
`npm start`, health check path `/healthz`, and the same three env vars set
by hand.

Note: Render's free tier spins the service down after 15 minutes idle, so
the first request after a quiet period takes ~30-60s to wake it back up.

## Auth

Email/password auth via [Better Auth](https://www.better-auth.com/). All
`/api/auth/*` routes are handled by its own router; the rest of the API
(`/api/bookings`, `/api/availability`) requires a valid session and derives
the user from the session cookie - there's no `playerName` field to trust
from the client anymore.

## API

- `GET /api/courts` - list courts (public)
- `GET /api/availability?courtId=&date=` - occupied `{hour, duration}` slots
  for a court/date, across all users, with no identifying info - used to
  grey out taken slots regardless of who booked them
- `GET /api/bookings` - the signed-in user's own bookings
- `POST /api/bookings` - create a booking (`courtId`, `date`, `hour`,
  `duration`) for the signed-in user; rejects overlapping slots with `409`
- `DELETE /api/bookings/:id` - cancel one of the signed-in user's own
  bookings (ownership is checked server-side)
